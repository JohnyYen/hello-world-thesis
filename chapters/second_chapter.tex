\chapter{\chapterSecond}

El presente capítulo presenta las decisiones de diseño y arquitectura que materializan los fundamentos pedagógicos descritos anteriormente en una plataforma educativa concreta. A partir de los principios del constructivismo, el aprendizaje basado en problemas y la teoría del flujo, se definen las mecánicas centrales del videojuego, el género narrativo adoptado y la historia que contextualiza los ejercicios de programación dentro de un entorno universitario reconocible para el estudiante.

Junto al módulo de juego, se detalla el diseño del tablero de analíticas del aprendizaje, componente que transforma las trazas generadas durante la experiencia de juego en información pedagógica accionable para el docente. La articulación entre ambos módulos (videojuego y tablero) define la propuesta central de la plataforma: cerrar el ciclo entre la experiencia del estudiante y la toma de decisiones del profesor a partir de datos estructurados y visualizaciones significativas.
Finalmente, el capítulo aborda los requisitos funcionales y no funcionales del sistema, los casos de uso identificados para cada actor, el modelo de datos, la arquitectura técnica de cada módulo y las tecnologías seleccionadas para su implementación. Cada elección tecnológica y arquitectónica se fundamenta en criterios directamente vinculados a los objetivos del proyecto, priorizando la coherencia entre las decisiones de diseño pedagógico y las decisiones de diseño de software.

\textbf{PostgreSQL} se selecciona para el backend por su cumplimiento nativo e incondicional del modelo ACID, indispensable para garantizar la integridad de operaciones críticas como el registro de progreso académico y la trazabilidad del desempeño~\cite{postgresql}. \textbf{SQLite} opera como motor embebido dentro del videojuego, almacenando localmente el estado de la sesión del jugador \cite{sqlite}. Su naturaleza de dominio público no introduce restricciones de licencia en el producto final, y los datos locales se sincronizan posteriormente con PostgreSQL para garantizar la coherencia entre la experiencia de juego y los registros académicos centralizados.

\section{Requisitos funcionales}

Los requisitos funcionales definen qué debe hacer el sistema desde el punto de vista del usuario o de otro sistema externo \cite{sommerville_software_2011,pressman_ingenierisoftware_2013}. En este caso se han dividido los requisitos funcionales según el módulo al que corresponde: \textit{videojuego} (ver Tabla \ref{tab:req-videogames}), \textit{aplicación web} (ver Tabla \ref{}), \textit{back-end} (ver Tabla \ref{}).

\requisitos{Requisitos funcionales del videojuego}{
	El sistema debe evaluar el desempeño del jugador en cada nivel utilizando métricas como tiempo, errores y eficiencia de la solución;
	El sistema debe registrar automáticamente los resultados obtenidos por el jugador para su análisis posterior;
	El sistema debe adaptar automáticamente la dificultad del nivel cuando el jugador no alcance los objetivos establecidos;
	La adaptación puede incluir la simplificación de objetivos, reducción de complejidad o provisión de pistas adicionales;
	Cada nivel debe estar asociado a un tema específico de programación claramente identificable;
	El sistema debe proporcionar retroalimentación visual o textual inmediata tras cada acción del jugador;
	El sistema debe ajustar la dificultad de los niveles futuros en función del historial de desempeño del jugador;
	El jugador debe poder configurar opciones del juego como sonido, resolución, idioma y modo de color;
	El sistema debe guardar automáticamente el progreso del jugador al finalizar o abandonar un nivel;
	Al inicio de cada nivel, el sistema debe mostrar una introducción teórica del concepto de programación asociado;
	El sistema debe permitir construir soluciones mediante bloques de programación visual;
	Los bloques de programación deben representar estructuras lógicas como condiciones, ciclos y variables;
	Al completar un nivel, el sistema debe mostrar una puntuación basada en criterios de rendimiento;
	El jugador debe poder consultar un resumen global de su progreso académico;
	Los niveles deben presentarse siguiendo un orden progresivo acorde al aprendizaje;
	El sistema debe restringir el acceso a niveles posteriores hasta completar los anteriores;
	El sistema debe ofrecer un tutorial interactivo en el primer inicio del videojuego;
	El jugador debe poder repetir niveles ya completados para mejorar su desempeño;
	El videojuego debe capturar automáticamente las interacciones del jugador (acciones sobre bloques, diálogos seleccionados, tiempos de respuesta) y transformarlas en declaraciones xAPI;
	El videojuego debe actualizar un modelo del estudiante a partir de sus interacciones, registrando aciertos, errores, tiempos y uso de ayudas;
	El videojuego debe presentar una narrativa interactiva en formato de novela visual que contextualice los ejercicios de programación dentro de una historia universitaria;
	El videojuego debe pausar el flujo narrativo para presentar la interfaz de programación cuando el nivel lo requiera;
	El videojuego debe desbloquear nuevos diálogos y avanzar la narrativa cuando el jugador resuelva correctamente el problema del nivel;
	El videojuego debe almacenar localmente las trazas xAPI cuando no exista conectividad y sincronizarlas con el servidor cuando se restablezca la conexión
}

Para ilustrar el uso de estos campos, se tomará como ejemplo el contexto de la biblioteca de la facultad, en el que el protagonista debe ayudar a un compañero a organizar una lista de libros siguiendo un criterio lógico establecido por el profesor. El campo \texttt{environment\_data} describe los elementos propios del entorno: la cantidad de estantes disponibles, los títulos presentes o el tipo de criterio de ordenamiento aplicable en esa escena. Finalmente, \texttt{defined\_actions} especifica las acciones concretas que pueden ejecutarse en el nivel; dado que otros niveles pueden reutilizar el contexto de la biblioteca, este campo permite restringir o ampliar el conjunto de acciones disponibles según las necesidades pedagógicas de cada uno.

\subsection{Patrones de diseño}

\textbf{Patrón singleton}


El patrón Singleton tiene como intención garantizar que una clase tenga únicamente una instancia globalmente accesible, previniendo inconsistencias que podrían surgir de crear múltiples instancias de componentes críticos. Este patrón resuelve específicamente el problema de gestionar estado compartido de forma segura en aplicaciones interactivas, donde múltiples instancias podrían causar conflictos de estado y comportamientos impredecibles.Como se aprecia en el diagrama del patrón Singleton (ver Figura \ref{fig:design-pattern-singleton}), la implementación en el proyecto Hello World utiliza la clase GameController como instancia única que centraliza la gestión del flujo general del juego y el estado global. Este enfoque resuelve el desafío de mantener consistencia en escenarios interactivos como cafetería o biblioteca, donde múltiples contextos podrían generar conflictos de estado. La implementación mejora significativamente la fiabilidad del motor de juego al proporcionar un único punto de control para el estado global, facilitando la depuración y previniendo errores de concurrencia que podrían surgir de múltiples instancias.

	\centering

	\caption[Diagrama del patrón Singleton]{Diagrama del patrón Singleton}
	\label{fig:design-pattern-singleton}
\end{figure}

\textbf{Patrón abstract factory}


El patrón Abstract Factory tiene como intención crear familias de objetos relacionados sin especificar sus clases concretas, permitiendo la creación flexible de objetos que deben trabajar juntos. Este patrón aborda específicamente el problema de la dependencia de clases concretas, facilitando la extensión y mantenimiento de sistemas que requieren diferentes configuraciones de objetos relacionados sin modificar el código existente.

Como se observa en el diagrama del patrón Abstract Factory (ver Figura \ref{fig:design-pattern-factory-method}), la aplicación en Hello World se centra en la creación dinámica de instancias de acciones dentro del bloque de ejecución. Este patrón resuelve el problema de generar acciones educativas de forma extensible, evitando la creación hardcoded de cada tipo de acción y permitiendo la adaptación del contenido educativo según las necesidades individuales de los estudiantes. La implementación facilita significativamente la personalización del aprendizaje al permitir agregar nuevos tipos de acciones sin alterar la lógica central del motor de ejecución.

\textbf{Patrón strategy}

El patrón Strategy tiene como intención encapsular algoritmos intercambiables y permitir su selección dinámica en tiempo de ejecución, facilitando la modificación de comportamientos sin alterar el código cliente. Este patrón resuelve específicamente el problema de la rigidez en la implementación de algoritmos, permitiendo adaptar el comportamiento del sistema a diferentes contextos sin modificar la lógica central.

Como se muestra en los diagramas del patrón Strategy aplicados al motor de ejecución (ver Figura \ref{fig:design-pattern-strategy-execution-engine}) y al agente de adaptación (ver Figura \ref{fig:design-pattern-strategy-adaptation-agent}), la implementación en Hello World presenta dos aplicaciones principales:

\subsection{Selección del género}

La elección del género de novela visual responde a evidencias empíricas que destacan su potencial diferenciado en motivación, retención del conocimiento y comprensión narrativa. A diferencia de géneros que priorizan la acción mecánica, la novela visual centra la experiencia en la historia y la toma de decisiones, lo que permite una alineación directa con objetivos pedagógicos de lectura e interpretación \cite{vn2}.

Desde la perspectiva motivacional, este género transforma el aprendizaje en una experiencia lúdica e interactiva que los estudiantes perciben más cercana al entretenimiento que al trabajo académico, incrementando su disposición activa, el interés y la persistencia \cite{vn1}. Esta mayor implicación emocional se traduce en una preferencia significativa por el formato frente a métodos tradicionales basados en texto plano, lo cual mitiga problemas de desmotivación en estudiantes con bajo compromiso inicial \cite{vn1}.

En términos cognitivos, la integración multimodal de texto, imagen e interactividad refuerza la codificación y recuperación de la información, facilitando la visualización de escenarios y la retención de elementos clave \cite{vn1}. Asimismo, al exigir que el estudiante participe activamente en la construcción del significado mediante decisiones narrativas, favorece el pensamiento crítico y una comprensión profunda de personajes, tramas y contextos. No obstante, aunque los resultados muestran una tendencia clara en la percepción positiva de los estudiantes, la ausencia de grupos de control y limitaciones metodológicas impiden afirmar que las novelas visuales sean superiores de manera concluyente a otros géneros o métodos \cite{vn3}; por tanto, se conciben como una herramienta complementaria que enriquece la experiencia educativa sin reemplazar necesariamente otras estrategias de enseñanza.

\subsection{Narrativa del juego}

El diseño narrativo contextualiza los conceptos abstractos de programación dentro de la vida académica real: el jugador, un estudiante de primer año de la Facultad de Ingeniería Informática, recibe misiones del profesor que exigen resolver problemas cotidianos de la facultad mediante lógica computacional. Esta ambientación genera empatía inmediata y activa el aprendizaje desde situaciones auténticas, alineándose con los principios del ABP. A lo largo de la trama, el protagonista cuenta con una mascota en forma de orca que actúa como guía durante la historia, mientras interactúa con compañeros y estudiantes avanzados.

La integración entre historia y mecánica se materializa en la estructura de niveles: cuando el profesor asigna una tarea, la novela visual pausa para dar paso a la interfaz de programación por bloques, donde el estudiante debe ensamblar la solución correspondiente al concepto enseñado. Si la respuesta es correcta, la narrativa avanza desbloqueando diálogos y progreso argumental; si es incorrecta, el sistema ofrece retroalimentación constructiva sin penalización, permitiendo reintentar y fomentando la autonomía. Desde el constructivismo, este ciclo obliga al estudiante a externalizar su modelo mental, ensayar y observar consecuencias inmediatas en un entorno seguro. Paralelamente, la progresión escalonada de la dificultad narrativa y técnica sostiene la teoría del flujo, manteniendo el desafío equilibrado con las habilidades adquiridas para evitar el aburrimiento inicial y la ansiedad en etapas avanzadas.

\subsection{Indicadores clave de rendimiento y aprendizaje}

Los indicadores se seleccionan según la disponibilidad de datos en las trazas
xAPI del videojuego y su pertinencia pedagógica para el seguimiento del
aprendizaje de la programación \cite{serrano2017xapi}, organizándose en tres
categorías:

\textbf{Indicadores de progreso académico:}
\begin{itemize}[itemsep=2pt, parsep=0pt]
	\item Niveles completados vs. niveles disponibles por concepto de programación.
	\item Porcentaje de superación por tema.
	\item Evolución histórica del desempeño individual a lo largo del tiempo,
	fomentando la autorregulación.
\end{itemize}

\textbf{Indicadores de desempeño en la resolución de problemas:}
\begin{itemize}[itemsep=2pt, parsep=0pt]
	\item Número de intentos por nivel y tasa de éxito en el primer intento.
	\item Tiempo promedio de resolución por nivel y por concepto.
	\item Tipos de errores más frecuentes, clasificados según el bloque lógico
	involucrado.
	\item Eficiencia de la solución: relación entre la cantidad de bloques utilizados
	y la solución óptima.
\end{itemize}

\textbf{Indicadores de motivación y flujo:}
\begin{itemize}[itemsep=2pt, parsep=0pt]
	\item Tiempo de sesión activa y frecuencia de abandono por nivel.
	\item Niveles repetidos voluntariamente, como indicador de motivación intrínseca.
	\item Estado de adaptación del agente: estrategia activa para cada estudiante
	(simplificación, complejización o neutro).
\end{itemize}

Cada interacción con los bloques, intento de solución y resultado de nivel
queda registrado como una declaración Actor-Verbo-Objeto que el backend
procesa y agrega para alimentar las visualizaciones del tablero, con soporte
para escenarios de conectividad intermitente.

El modelo de despliegue describe la distribución física y lógica de los componentes del sistema, especificando los nodos de ejecución, los canales de comunicación y los principales servicios externos involucrados durante la operación. Esta vista complementa la arquitectura lógica presentada en secciones anteriores, al evidenciar cómo se materializa la solución en una infraestructura concreta.

	\caption[Modelo de despliegue del sistema]{Modelo de despliegue del sistema}
	\label{fig:deployment-model}
\end{figure}

Como se observa en la Figura \ref{fig:deployment-model}, el sistema se organiza en una estructura cliente-servidor con separación explícita entre los clientes finales, la capa de presentación web y la capa de servicios del back-end. Esta distribución favorece la mantenibilidad, la escalabilidad y el control de las responsabilidades, al aislar los componentes que cambian con mayor frecuencia de aquellos que encapsulan la lógica de negocio y el acceso a datos.

En el extremo cliente se distinguen dos puntos de acceso principales. Por una parte, el módulo de videojuego educativo se ejecuta en el dispositivo del estudiante (por ejemplo, en entornos Windows o Android), donde se gestionan las interacciones en tiempo real, la presentación de escenas y la captura de acciones del usuario. Por otra parte, la aplicación web de gestión docente se ejecuta en el navegador del profesor, lo que permite un acceso multiplataforma sin requerir instalación local, y habilita la visualización de paneles analíticos y la administración del contenido educativo.

Ambos clientes consumen servicios expuestos por el back-end central mediante una API HTTP de estilo REST. Este canal de comunicación se establece sobre HTTPS, lo que contribuye a la confidencialidad e integridad de los datos intercambiados, particularmente en operaciones de autenticación, sincronización de progreso y consulta de métricas. Desde el punto de vista operacional, esta API actúa como punto único de entrada al dominio, normalizando las solicitudes y centralizando aspectos transversales como validación, autorización y registro de eventos.

El back-end se despliega como un servicio independiente en un servidor de aplicaciones (o contenedor mediante docker), donde residen las capas de casos de uso y adaptadores de interfaz. Esta ubicación concentra la lógica de negocio del sistema, permitiendo que las reglas del dominio se ejecuten de forma consistente para todos los clientes. Adicionalmente, el back-end integra mecanismos de persistencia, de modo que las operaciones de lectura y escritura se realizan contra una base de datos central, la cual almacena usuarios, cursos, niveles, progreso, métricas y registros históricos necesarios para el análisis pedagógico.

\section{Conclusiones parciales}

El presente capítulo concluye el diseño de la plataforma educativa \textit{Hello World}, abarcando desde las decisiones pedagógicas que fundamentan el videojuego hasta la arquitectura técnica que lo integra con el tablero de analíticas docente. Los resultados obtenidos permiten establecer las siguientes conclusiones:

\begin{itemize}
	\item Se seleccionó el género de novela visual como base del videojuego, sustentado en su capacidad para incrementar la motivación intrínseca y favorecer la retención del conocimiento, definiendo cuatro mecánicas centrales alineadas con los principios constructivistas y la teoría del flujo.
	
	\item Se estructuran tres vistas en el tablero de analíticas (general, individual y comparativa) alimentadas por trazas xAPI, donde cada componente visual responde a la naturaleza de los datos y al tipo de decisión pedagógica que debe facilitar.
\end{itemize}
