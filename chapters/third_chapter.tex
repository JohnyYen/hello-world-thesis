\section{Pruebas de caja negra}

El enfoque de pruebas de caja negra es empleado para validar el cumplimiento de los requisitos funcionales del sistema desde la perspectiva del usuario final, sin considerar la estructura interna del código fuente. Para la ejecución de esta etapa, se diseñaron casos de prueba enfocados en las dos vertientes principales de la solución: la experiencia de interacción directa dentro del videojuego y el flujo de procesamiento y visualización de datos en la plataforma web educativa.

En primer lugar, se evaluó el núcleo de la experiencia lúdica mediante los casos de prueba detallados en la Tabla~\ref{tab:cp-playgame} y la Tabla~\ref{tab:cp-setconfig}. Estos escenarios permiten verificar que la carga de los niveles a partir de los archivos de configuración se realiza de forma íntegra y que el jugador tiene el control total sobre las variables de personalización del entorno. Validar este aspecto resulta crítico para asegurar que las mecánicas de juego no se vean interrumpidas por fallos en la inicialización o en la aplicación de la configuración global.

\section{Pruebas unitarias}

El objetivo principal fue asegurar que cada módulo por separado produjera los resultados esperados frente a diferentes entradas, sin depender de sus componentes vecinos. Dentro de esta etapa, se prestó especial atención al intérprete de soluciones; se verificó que el módulo encargado de analizar las acciones y disposiciones de bloques realizadas por el estudiante fuera capaz de evaluar de forma exacta y determinista si la solución planteada resuelve el problema del nivel, garantizando la precisión en la retroalimentación inmediata del juego.

\section{Pruebas de integración}

Una vez validados los componentes de forma aislada, se procede a diseñar las pruebas de integración. El propósito de esta etapa es determinar que la comunicación y el intercambio de datos entre los distintos módulos se realizan de forma correcta y sin pérdida de información. Para ello, se evaluan tres flujos fundamentales del sistema: en primer lugar, la adaptación de niveles, verificando que el agente de adaptación pudiera leer e interpretar correctamente los archivos de configuración para construir el entorno de forma íntegra; en segundo lugar, el flujo de conversión de datos a estadísticas, asegurando que las interacciones brutas del jugador fueran transformadas adecuadamente en métricas comprensibles; y finalmente, la sincronización, validando que la comunicación entre el videojuego y el servidor central fluyen sin errores de formato, asegurando que los \textit{batch} de datos lleguen intactos para su posterior almacenamiento.

\section{Pruebas de seguridad}

Dado que la aplicación gestiona datos académicos y telemetría de estudiantes (a través de declaraciones xAPI), la integridad y privacidad de la información son aspectos críticos. Por ello, se diseña un conjunto de pruebas de seguridad no funcionales. Estas evaluaciones se centraron en identificar vulnerabilidades comunes, tales como intentos de inyección de código o modificaciones maliciosas en los archivos de configuración locales (JSON) antes de ser procesados por el motor. Adicionalmente, se evaluó la robustez del protocolo de sincronización para garantizar que la transmisión de datos hacia el servidor no fuera susceptible de interceptaciones (Man-in-the-Middle) ni de manipulación de payloads, asegurando que solo las trazas legítimas generadas por el flujo authéntico del juego sean registradas en la base de datos.
